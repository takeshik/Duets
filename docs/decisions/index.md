| [ADR-4](4_use-jint-as-the-javascript-engine.md) | Use Jint as the JavaScript Engine | Jint, JavaScript engine, CLR interop, AOT, pure .NET, sandboxing | Confirms Jint over V8/ClearScript, Node.js hosting, and other managed engines; its pure C# implementation satisfies the universal embeddability requirement, while strong CLR interop (AllowClr) and built-in sandboxing meet the remaining drivers. | [ADR-5](5_separate-jint-engines-for-typescript-compiler-and-user-code.md) | Separate Jint Engines for TypeScript Compiler and User Code | Jint, engine isolation, TypeScriptService, ScriptEngine, AllowClr, security | Runs the TypeScript compiler and user scripts in two independent Jint engine instances; isolation prevents user code from polluting compiler global state, and each engine requires a different security policy (compiler sandboxed; user engine uses AllowClr). |
| [ADR-6](6_fetch-and-cache-runtime-js-assets-from-cdn.md) | ~~Fetch and Cache Runtime JS Assets from CDN~~ *(Deprecated — superseded by ADR-18)* | CDN, unpkg, caching, typescript.js, monaco, assembly size | Originally established fetching typescript.js and monaco loader.js from unpkg at runtime and caching in the system temp directory for 7 days; superseded by ADR-18, which introduces a pluggable IAssetSource abstraction while preserving this default. |
| [ADR-7](7_use-monaco-editor-as-the-browser-based-repl-ui.md) | Use Monaco Editor as the Browser-based REPL UI | Monaco, REPL, browser, VS Code, embedded resources, completions | Selects Monaco Editor over a VS Code extension approach; Monaco runs in-browser with no external process or installation, provides TypeScript-aware editing out of the box, and is served as embedded static resources within the library assembly. |
| [ADR-8](8_use-addextralib-to-inject-dts-declarations-for-completions.md) | Use addExtraLib to Inject .d.ts Declarations for Completions | Monaco, addExtraLib, d.ts, completions, ClrDeclarationGenerator, SSE | Uses Monaco's built-in TypeScript language service with addExtraLib over external tsserver or Jint runtime inspection; ClrDeclarationGenerator produces .d.ts from .NET types via reflection and declarations are pushed to Monaco via SSE. | [ADR-9](9_wrap-httplistener-in-a-dedicated-middleware-library.md) | Wrap HttpListener in a Dedicated Middleware Library | HttpHarker, middleware, HttpListener, composability, separation of concerns | Extracts HTTP infrastructure into a separate middleware-based library (HttpHarker) with no dependency on Duets; raw HttpListener usage did not scale for multiple web interfaces, and the Rin pattern of serving UI as composable middleware inspired the design. |
| [ADR-10](10_extract-itranspiler-interface-for-scriptengine.md) | Extract ITranspiler Interface for ScriptEngine | ITranspiler, ScriptEngine, TypeScriptService, interface, dependency injection, testability | Introduces ITranspiler as the minimal interface between ScriptEngine and its transpiler; ScriptEngine requires it at construction and always transpiles before executing, encapsulating the transpile-then-execute pattern and enabling stub transpilers in tests without the full TypeScript compiler. |
| [ADR-11](11_sandbox-multi-mode-debugging-cli.md) | Sandbox as a Multi-Mode Debugging CLI *(Partially superseded by ADR-16)* | Duets.Sandbox, batch mode, JSONL, AI agent, REPL, CLI | Transforms Duets.Sandbox into a multi-mode CLI (REPL, eval, complete, serve, batch); the JSONL batch mode lets AI coding agents pipe operations to stdin and read results from stdout without background process management, with state persisting across operations within a session. |
| [ADR-12](12_language-service-host-rewrite-and-nolib.md) | Rewrite LanguageServiceHost and Provide Standard Library via CDN | LanguageServiceHost, language-service.js, lib.es5.d.ts, completions, InjectStdLibAsync | Fully rewrites the TypeScript LanguageServiceHost to implement all methods required by createLanguageService, fixing a crash on repeated GetCompletions calls; adds opt-in injection of lib.es5.d.ts from CDN via InjectStdLibAsync() to enable completions for JavaScript built-ins alongside registered .NET types. |
| [ADR-13](13_script-built-ins-and-typings-object.md) | Script Built-ins and the `typings` Object | typings, ScriptBuiltins, ScriptTypings, RegisterTypeBuiltins, useNamespace, scanAssembly | Introduces the typings global object (useType, scanAssembly, scanAssemblyOf, useAssembly, useAssemblyOf, useNamespace) in the Duets core library, replacing per-host importTypeDefs wiring and making type-declaration management reusable across any embedding application. |
| [ADR-14](14_assembly-derivation-from-type-reference.md) | Assembly Derivation from TypeReference via `scanAssemblyOf` / `useAssemblyOf` | scanAssemblyOf, useAssemblyOf, TypeReference, AllowSystemReflection, typings API | Adds scanAssemblyOf and useAssemblyOf to ScriptTypings, deriving the assembly from a CLR TypeReference in C# rather than in script, sidestepping the AllowSystemReflection requirement while keeping the call ergonomic when a type reference is already in scope. |
| [ADR-15](15_namespace-skeleton-dummy-member.md) | Namespace Skeleton Visibility via `$name` Dummy Member | scanAssembly, namespace skeleton, $name, TypeScript language service, completions | Resolves TypeScript's filtering of empty declare namespace blocks by injecting a `const $name` dummy member into skeletons produced by scanAssembly; the member is automatically removed (skeleton overwritten with empty declaration) when a real type for that namespace is registered. |
| [ADR-16](16_samples-directory-and-sandbox-role-clarification.md) | Introduce `samples/` Directory and Clarify Duets.Sandbox Role | samples, file-based apps, Duets.Sandbox, dotnet run, role clarification | Introduces a samples/ directory of self-contained file-based .cs examples (dotnet run samples/<file>.cs), separating usage examples from internal tooling; simultaneously removes the eval command from Duets.Sandbox so all remaining commands use the fully-initialized SandboxSession uniformly. |

| [ADR-18](18_pluggable-asset-source-abstraction.md) | Pluggable Asset Source Abstraction (`IAssetSource`) | IAssetSource, AssetSources, asset fetching, CDN, offline, testability, HttpClient | Lifts the YAGNI deferral from ADR-6 by introducing IAssetSource and AssetSources factory to make runtime JS asset fetching pluggable (HTTP, unpkg, embedded resources, delegates, composable disk cache); default behavior is identical to ADR-6, but consumers in air-gapped environments can now supply assets without forking the library. |
| [ADR-19](19_babel-transpiler-as-typescript-7-migration-path.md) | Babel Transpiler as TypeScript 7 Migration Path | BabelTranspiler, Babel, @babel/standalone, TypeScript 7, Go rewrite, ITranspiler, migration | Introduces BabelTranspiler — an ITranspiler implementation backed by @babel/standalone running in Jint — as the forward-compatibility path for TypeScript 7, which ships only a native Go binary with no JS API; chosen over ts-blank-space (no enum/namespace support), sucrase (no standalone bundle, broken namespace output), and TypeScript 6.x freeze (dead end). |
| [ADR-20](20_dump-as-global-function-not-prototype-extension.md) | `dump` Implemented as a Generic Global Function, Not a Prototype Extension | dump, Object.prototype, Monaco, addExtraLib, interface Object, generics, completions | Implements `dump<T>(value: T): T` as a global function rather than an `Object.prototype` extension because Monaco's `addExtraLib` does not propagate `interface Object` augmentations to specific inferred types, making prototype-based completion discovery impossible without losing type precision. |
| [ADR-21](21_typings-import-namespace.md) | `typings.importNamespace` — Namespace Import with Completion Registration | typings, importNamespace, AllowClr, NamespaceReference, Jint internals, completions | Adds `typings.importNamespace(ns)` to combine runtime namespace access and TypeScript completion registration in one call; deliberately leaves `globalThis.importNamespace` unchanged because Jint registers it as non-writable/non-configurable and overriding it requires non-public Jint APIs whose stability cannot be guaranteed. |
